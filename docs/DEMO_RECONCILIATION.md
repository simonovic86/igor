# Bridge Reconciliation Agent: Fail, Migrate, Resume, Prove

**Status:** Pre-build spec
**Purpose:** First product demo — demonstrate Igor's value to a real buyer persona

## Why This Demo

Igor's architecture is strong. The missing piece is a visceral demonstration that connects survivability, migration, and replay to a workflow someone would pay for.

This demo is not a showcase of features. It is a proof that Igor solves a painful, expensive problem: ambiguous state and duplicate execution in long-lived financial automation.

### Target audience

| Role | What they need to see |
|------|----------------------|
| Engineer | Deterministic replay, correct state recovery, no duplicate side effects |
| Operator | Human-readable incident timeline, clear answer to "what happened at 3am" |
| Manager / Compliance | Proof that no action ran twice, auditable narrative |

## Demo Title

**Bridge Reconciliation Agent: Fail, Migrate, Resume, Prove**

## What It Must Show (Minimum Viable Demo)

All of these. If any is missing, the punch gets softer.

1. Agent detects a pending transfer case
2. Agent advances through a small state machine
3. Agent performs one simulated external side effect (finalize)
4. Host process dies at a bad moment (after intent, before ack)
5. Agent migrates to a peer node and resumes
6. Finalize is **not** executed twice
7. Replay generates a human-readable timeline
8. Timeline explicitly explains **why** duplicate execution did not happen

## Agent State Machine

```
DetectedPendingTransfer
        │
        ▼
WaitingConfirmations
        │
        ▼
ReadyToFinalize
        │
        ▼
   Finalized
        │
        ├──▶ Disputed
        │        │
        │        ▼
        │    Escalated
        │
        ▼
    Completed
```

### States

| State | Description |
|-------|-------------|
| `DetectedPendingTransfer` | Agent picks up a new cross-chain transfer case |
| `WaitingConfirmations` | Waiting for source-chain confirmation threshold |
| `ReadyToFinalize` | Confirmations met, ready to execute finalize on destination chain |
| `Finalized` | Finalize side effect committed |
| `Disputed` | Anomaly detected post-finalize (optional for v1) |
| `Escalated` | Dispute escalated to human review (optional for v1) |
| `Completed` | Case fully resolved |

## Failure Injection Points

These are the moments where real money gets lost or doubled in production:

| Point | What breaks | What Igor must do |
|-------|------------|-------------------|
| After marking local state `ReadyToFinalize`, before external call | State says "ready" but call never happened | Resume and execute finalize exactly once |
| After finalize call intent recorded, before acknowledgment | Did it go through? Unknown | Resume, detect intent was recorded, skip duplicate, wait for ack |
| During checkpoint write | Partial state on disk | Atomic checkpoint prevents corruption; resume from last good checkpoint |
| During migration handshake | Agent in-flight between nodes | Single-instance invariant (EI-1) prevents both nodes from ticking |

## Side-by-Side Contrast

The demo should include a brief "what usually happens" baseline. Not a full competitor benchmark — just enough to make Igor's value legible in one glance.

### Naive Worker (typical automation)

```
03:14:07  Finalize called
03:14:08  Process crashed
03:14:11  Process restarted
03:14:11  State unclear — did finalize go through?
03:14:11  Options:
          a) Retry finalize → risk: double execution
          b) Skip finalize → risk: stuck transaction
          c) Wake up human → cost: hours of investigation
```

### Igor Worker

```
03:14:07  Finalize intent recorded with idempotency key
03:14:08  Node failed
03:14:11  Agent migrated, resumed from checkpoint
03:14:12  Replay verified — intent already committed, skip duplicate
03:14:13  Ack received, case completed
03:14:14  Timeline generated — full incident proof
```

## Scope Rules

### In scope for v1

- [ ] Reconciliation agent with state machine (5 core states)
- [ ] Simulated external side effect with idempotency key tracking
- [ ] Failure injection at configurable points
- [ ] Migration to second node and resume
- [ ] No-duplicate-execution proof
- [ ] Human-readable timeline output from replay engine
- [ ] Side-by-side comparison script or output

### Out of scope for v1

- Real blockchain integration (simulated is fine)
- Disputed/Escalated states (defer to v2)
- Multiple concurrent cases (one case is enough)
- Performance benchmarks
- UI/dashboard (terminal output is the product for now)

## Ruthless Rule

> No new architecture work until the reconciliation demo can fail on purpose and explain itself to a human.

## Success Criteria

The demo succeeds when someone watching it says:

*"I have this exact problem. When can I use this?"*

Not:

*"Cool technology."*

If the reaction is the second one, the demo is not done yet.
